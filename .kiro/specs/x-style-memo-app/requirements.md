# Requirements Document

## Project Description (Input)
# X(Twitter)風メモアプリ 仕様書

> 個人用・ローカル完結・iOS(SwiftUI) のショートメモアプリ ＋ オンデバイスAIによる擬似SNS体験

---

## 1. 概要

Xのタイムライン体験（短文を逆時系列で書き溜める）を、**自分一人のためのメモ帳**として再構成したiOSアプリ。サーバーを持たず、すべてのデータは端末内で完結する。

さらに本アプリは、他者がいない個人メモに「SNSの手触り」を与えるため、**オンデバイスAI（Apple Foundation Models）が生成するAIペルソナ**を導入する。AIペルソナがあなたの投稿に「いいね」したり「リプライ」したりすることで、投稿して反応が返ってくるフィードバックループを、ネットワークなし・端末内で再現する。

「X風」とは見た目の模倣ではなく、**「短文を気軽に放り込み、時系列で振り返り、反応が返ってくる」という入力・閲覧・反応の流儀**を指す。実在の他者要素（フォロー・リツイート・通知の外部送信）は持たない。

## 2. 技術スタック

| 項目 | 採用 | 理由 |
| --- | --- | --- |
| プラットフォーム | iOS 17+（コア） / iOS 26+（AI機能） | SwiftDataは iOS 17+、Foundation Models は iOS 26+ かつ Apple Intelligence 対応端末が必要 |
| UI | SwiftUI | 宣言的UI。`@Query` でデータと画面を直結できる |
| 永続化 | SwiftData | `@Model` でモデル定義、Core Dataより記述量が少ない |
| 画像保存 | SwiftData `@Attribute(.externalStorage)` | 大きな `Data` をDB本体から自動で外部ファイルへ退避し、DBの肥大化を防ぐ |
| 画像選択 | PhotosUI (`PhotosPicker`) | 標準フォトピッカー |
| AI（生成） | Foundation Models（オンデバイス System Language Model） | APIキー・ネットワーク・課金なし。`@Generable` で構造化出力。「ローカルのみ」方針を崩さない |
| AI（関連メモ検索 / RAG） | Foundation Models の Spotlight 検索ツール（iOS 27+） | 端末のSpotlightインデックスを使う完全ローカルのRAG。代替経路は §8.5 参照 |

**なぜオンデバイスAIなのか**：「ローカルのみ」を守るため、クラウドモデル（Claude/Gemini）やPrivate Cloud Computeは今回採用しない。オンデバイスのSystem Language Modelはネットワークもキーも課金も不要で、データが端末外に出ないため、本アプリの哲学と完全に一致する。クラウドモデルへの差し替えは将来の拡張候補（§10）に置く。

## 4. 機能スコープ

### 4.1 含む機能（コア）

- **メモ投稿**：テキストを入力して保存する
- **タイムライン表示**：自分のルートメモを逆時系列で一覧表示
- **削除**：メモを削除する
- **ハッシュタグ**：本文中の `#xxx` を自動抽出し、タグで分類・絞り込み
- **本文検索**：入力文字列で本文を部分一致検索
- **スレッド**：あるメモへの返信を連ね、親子として表示
- **画像添付**：メモに画像を1枚添付（端末内に保存）

### 4.2 含む機能（AI上乗せレイヤー）

> いずれもオンデバイスAIで実現。Apple Intelligence非対応端末では自動的に非表示となり、コア機能のみで動作する（縮退）。

- **AIいいね**：投稿に対し、ペルソナがいいねを付ける
- **AIリプライ**：ペルソナがスレッドに返信を生成する
- **AIペルソナ**：性格を持つ複数のAIキャラクター。各自の口調で反応する
- **関連メモの提示（RAG）**：過去メモから関連するものを検索し、ペルソナが「気づき」として提示する

### 4.3 見送る機能（およびその理由）

| 機能 | 見送り理由 |
| --- | --- |
| 編集 | メモを投稿後イミュータブルとする割り切り。修正は「削除→再投稿」 |
| ブックマーク | 単独利用では優先度低と判断 |
| リツイート | 他者が存在しないため意味を持たない |
| フォロー / 外部通知 | 同上。AIの反応はアプリ内で完結 |
| 認証 | 単一ユーザー・ローカル完結のため不要 |
| クラウドAI / サーバー同期 | 「ローカルのみ」方針のため対象外（将来候補） |

> **編集を外したことの含意**：メモは投稿後に変更不可。データモデルから `updatedAt` を省略できる。将来編集を足す場合は `updatedAt` 列の追加で対応する。

TabView
├─ ホームタブ
│   └─ NavigationStack
│       ├─ タイムライン画面 ………… 自分のメモ一覧（逆時系列）
│       │   └─ コンポーザー画面 …… 新規投稿 / 返信投稿

│   └─ NavigationStack
│       └─ 検索画面 ……………… 本文検索 + タグ絞り込み

    └─ NavigationStack
        └─ アクティビティ画面 …… 誰が・いつ・どの投稿に反応したかの一覧

X風の体験に合わせ、最下部にタブを置く。AI機能が利用不可の端末では「アクティビティタブ」を表示しない。投稿は画面右下のフローティングボタン（FAB）から開始する。

## 6. 画面ごとの詳細仕様

### 6.1 タイムライン画面（ホーム）

- 自分のルートメモ（`author == nil` かつ `parent == nil`）を `createdAt` の降順で一覧表示する。
- 各セルの表示内容：
  - 本文（タグ部分 `#xxx` はリンク風に強調）
  - 相対時刻（例：「3分前」「昨日」）
  - 画像があればサムネイル
  - いいね数（`likedBy.count`）／リプライ件数（スレッドがある場合）
- 右下のFABタップ → コンポーザー画面（新規投稿）。
- セルタップ → スレッド詳細画面。
- セルの左/右スワイプ → 削除（確認ダイアログあり）。

### 6.2 コンポーザー画面（投稿 / 返信）

- 複数行テキスト入力欄。
- 文字数カウンタを表示（上限は設けない。現在の文字数のみ表示）。
- 画像添付ボタン（`PhotosPicker`）。添付は1枚まで。添付後はプレビューと削除ボタンを表示。
- 「投稿」ボタンで保存し、前の画面へ戻る。保存後、AIリアクション生成を非同期で開始する（§8.6）。
- 返信モードで開いた場合（親メモIDを受け取った場合）は、保存時にそのメモを親として紐づける。

### 6.3 スレッド詳細画面

- 親メモを上部に表示。
- その下にリプライ群（自分のもの／ペルソナのもの）を `createdAt` の昇順で表示。ペルソナのリプライはペルソナ名・アバターを添えて区別する。
- 関連メモの提示があれば、ペルソナの「気づき」として表示し、参照元メモへ遷移できる（§8.5）。
- 下部の「返信」導線 → コンポーザー画面（返信モード、親メモIDを渡す）。
- 親メモ削除時の挙動 → 6.5 削除仕様 参照。

### 6.4 検索画面

- テキスト入力で本文を部分一致検索（初期は単純な `contains`）。
- タグ一覧を表示し、タグをタップするとそのタグを持つメモのみに絞り込む。
- 検索結果は逆時系列で表示。

### 6.5 削除仕様

- 通常メモ：対象メモを物理削除。`likedBy` の関連は解除される。
- スレッドの親メモ：確認ダイアログを表示し、了承時は**リプライ（自分・AI問わず）も連鎖削除（cascade）**する。`replies` リレーションに `.cascade` の削除ルールを設定する。
- 削除前に確認ダイアログを必ず挟む（編集機能が無いぶん削除が取り返しのつかない操作になるため）。

### 6.6 アクティビティ画面（AI機能有効時のみ）

- ペルソナが行ったリアクション（いいね・リプライ・気づき）を新しい順に一覧表示する。
- 各行タップで該当メモ／スレッドへ遷移。
- 「投稿したら反応が返ってくる」体験の受け皿。Xの通知タブに相当する役割。

## 7. データモデル（SwiftData）

### 7.1 エンティティ

#### `Memo`

| プロパティ | 型 | 説明 |
| --- | --- | --- |
| `id` | `UUID` | 一意識別子 |
| `body` | `String` | 本文 |
| `createdAt` | `Date` | 作成日時。並び順の基準 |
| `imageData` | `Data?` | 添付画像。`@Attribute(.externalStorage)` |
| `author` | `Persona?` | 作者。`nil` なら自分、非nilならそのペルソナのリプライ |
| `parent` | `Memo?` | 返信先の親メモ（無ければ `nil` = ルート投稿） |
| `replies` | `[Memo]` | このメモへの返信群（`parent` の逆関係、削除ルール `.cascade`） |
| `likedBy` | `[Persona]` | いいねしたペルソナ群（多対多）。いいね数 = `likedBy.count` |
| `tags` | `[Tag]` | 紐づくタグ（多対多） |

#### `Persona`

| プロパティ | 型 | 説明 |
| --- | --- | --- |
| `id` | `UUID` | 一意識別子 |
| `name` | `String` | 表示名（例：「きょうこ」「博士」） |
| `personality` | `String` | 性格を表す指示文。`LanguageModelSession` の `instructions` に渡す |
| `accentColor` | `String` | アバター・名前の表示色（識別用） |
| `authoredMemos` | `[Memo]` | このペルソナが書いたリプライ（`Memo.author` の逆関係） |
| `likedMemos` | `[Memo]` | このペルソナがいいねしたメモ（`Memo.likedBy` の逆関係） |

#### `Tag`

| プロパティ | 型 | 説明 |
| --- | --- | --- |
| `name` | `String` | タグ名（先頭の `#` を除いた文字列。重複なし） |
| `memos` | `[Memo]` | このタグを持つメモ群（`Memo.tags` の逆関係） |

### 7.2 設計判断のメモ

- **AIリプライを別概念にしない**：リプライはすべて `Memo` であり、`author` が非nilかどうかで「自分の返信」か「ペルソナの返信」かを区別するだけ。スレッド表示ロジックを一本化できる。
- **タグを別エンティティにする理由**：投稿時にタグを抽出して `Tag` として実体化しておけば、タグ一覧・件数・絞り込みがリレーション辿りで安価に実現できる。
- **いいねを `likedBy` 関連で表す理由**：単なるカウンタにせず「誰がいいねしたか」を持つことで、アクティビティ画面や「○○さんといいね」表示が可能になる。数が必要なときは `likedBy.count`。
- **画像の外部保存**：`@Attribute(.externalStorage)` により大きな `Data` をストア外へ自動退避し、一覧取得時のメモリ効率を保つ。

## 8. 主要な処理仕様

### 8.1 ハッシュタグ抽出

- メモ保存時に本文を走査し、ハッシュタグを抽出する。
- 抽出パターン：`#` の直後に続く「文字・数字・アンダースコア」の連なり。
  - 正規表現の文字クラスは `\p{L}\p{N}_` を用いる（`\p{L}` はあらゆる言語の文字を表すUnicodeプロパティ。`#メモ` のような日本語タグも抽出できる。`[a-zA-Z0-9]` だと日本語が漏れる）。
- 抽出した各文字列について、同名の `Tag` が既にあれば再利用し、無ければ新規作成して `Memo.tags` に紐づける（upsert）。

### 8.2 検索

- 本文検索：入力文字列で `Memo.body` を部分一致（`contains`）でフィルタ。
- タグ絞り込み：選択した `Tag` に紐づく `memos` を表示。
- いずれの結果も `createdAt` 降順で並べる。

### 8.3 AIペルソナ

- アプリ初回起動時に、プリセットのペルソナを数体シードする（例：共感系・ツッコミ系・専門家系）。`personality` は性格を表す短い指示文として保持する。
- 各ペルソナの反応生成は、そのペルソナの `personality` を `LanguageModelSession(instructions:)` に渡した独立セッションで行う。
- ペルソナは「AIキャラクターである」ことが分かる形で提示する（実在の他者を装わない）。これは欺瞞を避けるためのプロダクト上の方針。

### 8.4 AIいいね / AIリプライ（構造化出力）

- なぜ自由文でなく構造化出力か：「いいねするか」「返信するか」「返信文」を型で受け取り、後続処理を分岐ひとつで書けるようにするため。
- 反応の型（`@Generable`）：

```swift
import FoundationModels

@Generable
struct PersonaReaction {
    @Guide(description: "この投稿にいいねするか")
    let shouldLike: Bool

    @Guide(description: "返信するか")
    let shouldReply: Bool

    @Guide(description: "返信する場合の短いカジュアルな本文。しない場合は空")
    let replyText: String
}

func react(to memo: Memo, as persona: Persona) async throws -> PersonaReaction {
    let session = LanguageModelSession(instructions: persona.personality)
    let response = try await session.respond(
        to: "次の投稿に反応して：\(memo.body)",
        generating: PersonaReaction.self
    )
    return response.content
}
```

- 適用：`shouldLike` が真なら `memo.likedBy` にペルソナを追加。`shouldReply` が真なら `Memo(author: persona, parent: memo, body: replyText)` を作成してスレッドに挿入。

### 8.5 関連メモの提示（RAG）

- 目的：新規投稿に関連する過去メモを見つけ、ペルソナが「これは先月の○○と繋がっていますね」という気づきとして提示する。
- 本命経路（iOS 27+）：Foundation Models の **Spotlight 検索ツール**を利用する。各メモを Core Spotlight に `CSSearchableItem` としてインデックスしておき、モデルがSpotlight経由で関連メモを取得して文脈に取り込む（完全オンデバイスのRAG）。
- 代替経路（iOS 17+、AI不要）：
  - タグ重複によるスコアリング（共有タグ数が多いメモを関連とみなす）。安価で決定的。
  - `NaturalLanguage` の `NLContextualEmbedding` で本文をベクトル化し、コサイン類似度で近いメモを抽出。
- いずれの経路でも、取得した関連メモを `PersonaReaction` 生成時の文脈に与えれば、「気づき」は実際の過去メモに基づいた（grounded）生成になる。これがRAGたるゆえん。

### 8.6 生成タイミングと並行性

- 投稿は即座に保存し、AIリアクションは**非同期で後から生成**する。これにより投稿時にUIが固まらず、反応がパラパラ届く本物のSNSに近い体験になる。
- 各ペルソナの反応は、わずかな遅延（`Task.sleep`）を挟んで時間差で反映する。
- オンデバイスモデルは共有資源のため、セッションは過度に並行させず、逐次または低い並行度で実行する。

### 8.7 画像保存

- `PhotosPicker` で選択した画像を `Data` に変換し、`Memo.imageData` に格納（外部保存属性により実体は外部ファイル）。
- 1メモにつき1枚まで。

## 9. 非機能要件

- **対応OS**：コア機能は iOS 17 以降。AIいいね/リプライは iOS 26 以降かつ Apple Intelligence 対応端末。関連メモのSpotlight RAGは iOS 27 以降。
- **縮退動作**：起動時に `SystemLanguageModel` の可用性を確認し、利用不可ならAI機能（アクティビティタブ・AIリアクション）を非表示にして、純粋なメモアプリとして動作する。
- **オフライン**：ネットワーク不要。AIを含む全機能が常時オフラインで動作する。
- **データの所在**：アプリのサンドボックス内。AI推論も端末内で完結し、データは端末外に出ない。明示的な同期は行わない。
- **パフォーマンス**：メモ件数が数千件規模でもタイムラインのスクロールが滑らかであること（`@Query` のソート・フィルタはDB側で行う）。AI生成はバックグラウンドで行いUIをブロックしない。

## Introduction

本仕様は、iOS アプリ **Tegaru** の要件を定義する。Tegaru は、X(Twitter) のタイムライン体験を「自分一人のためのメモ帳」として再構成したアプリである。サーバーを持たず、すべてのデータは端末内（アプリのサンドボックス内）で完結する。コア機能（メモの投稿・閲覧・編集・削除・タグ・検索・スレッド・画像添付）に加え、Apple Foundation Models によるオンデバイス AI ペルソナが投稿に「いいね」「リプライ」「関連メモの気づき」を返すことで、ネットワークなしで擬似的な SNS のフィードバックループを再現する。AI 機能は対応端末でのみ有効化され、非対応端末では自動的に縮退して純粋なメモアプリとして動作する。

**「X 風」の定義（重要）**: Tegaru における「X 風」とは、X の見た目（ロゴ・配色・UI レイアウト）や SNS としての機能（フォロー・リツイート・外部通知・他ユーザーとの交流）を模倣することを意味しない。あくまで「短文を気軽に放り込み、時系列で振り返り、反応が返ってくる」という **入力・閲覧・反応の“流儀”** だけを借りる。実在の他者が関与する要素は一切持たず、反応はすべて端末内の AI ペルソナによりアプリ内で完結する。

本文書は「何を満たすか（WHAT）」を EARS 形式で定義する。実装方法（HOW）は設計フェーズで扱う。

## Boundary Context

- **In scope**: メモの投稿／逆時系列タイムライン表示／メモ編集（本文＋画像、`updatedAt` 記録と「編集済み」表示）／削除（cascade 含む）／ハッシュタグ抽出と絞り込み／本文・タグ検索／スレッド（返信の親子表示）／画像1枚添付／AIペルソナのシードと提示／AIいいね・AIリプライ（構造化出力・非同期生成）／関連メモ提示（RAG）／アクティビティ一覧／AI可用性に応じた縮退／ローカル完結のデータ永続化
- **Out of scope**: ブックマーク／リツイート／フォロー・外部通知／認証／クラウドAI・サーバー同期・iCloud同期／ユーザーによるペルソナのカスタム作成（いずれも §10 の将来拡張候補）／X の視覚的・ブランド的模倣（ロゴ・配色・UIレイアウトの丸写し）／実在の他者が関与するあらゆる SNS 要素
- **Adjacent expectations**: SwiftData（iOS 17+）、PhotosUI（`PhotosPicker`）、Foundation Models（iOS 26+ かつ Apple Intelligence 対応端末）、Core Spotlight + Foundation Models Spotlight 検索ツール（iOS 27+）、NaturalLanguage（代替RAG）

## Requirements

### Requirement 1: メモ投稿（コンポーザー）
**Objective:** ユーザーとして、短文テキストを素早く入力して保存したい。それによって、思いついたことを気軽に書き溜められる。

#### Acceptance Criteria
1. When ユーザーがコンポーザー画面で本文を入力し「投稿」を実行したとき、コンポーザーは入力本文を `body`・現在時刻を `createdAt` とする新しいメモを永続化しなければならない（shall）。
2. While ユーザーが本文を入力している間、コンポーザーは現在の文字数を表示しなければならない（shall）。
3. The コンポーザーは本文の文字数に上限を設けてはならない（shall not）。
4. If 本文が空（空白文字のみを含む）の状態で投稿が実行されたとき、then コンポーザーはメモを保存せず投稿操作を無効化しなければならない（shall）。
5. When メモの保存が完了したとき、コンポーザーは直前の画面へ自動的に戻らなければならない（shall）。
6. When メモの保存が完了したとき、コンポーザーは AI リアクション生成を非同期に開始しなければならない（shall）。ただし AI 機能が利用可能な場合に限る。
7. Where 返信モードで起動された（親メモIDが渡された）場合、コンポーザーは保存するメモの `parent` を当該親メモに設定しなければならない（shall）。

### Requirement 2: タイムライン表示
**Objective:** ユーザーとして、自分のメモを新しい順に一覧で振り返りたい。それによって、過去の記録を時系列で見渡せる。

#### Acceptance Criteria
1. The タイムラインは `author == nil` かつ `parent == nil` を満たすメモ（自分のルートメモ）のみを一覧表示しなければならない（shall）。
2. The タイムラインはメモを `createdAt` の降順（新しい順）で並べなければならない（shall）。
3. The タイムラインは各メモについて、本文・相対時刻・いいね数（`likedBy.count`）を表示しなければならない（shall）。
4. Where メモに画像が添付されている場合、タイムラインはサムネイルを表示しなければならない（shall）。
5. Where メモに1件以上の返信がある場合、タイムラインはリプライ件数を表示しなければならない（shall）。
6. The タイムラインは本文中のハッシュタグ（`#xxx`）部分をリンク風に強調表示しなければならない（shall）。
7. When ユーザーがメモセルをタップしたとき、ホームナビゲーションは当該メモのスレッド詳細画面へ遷移しなければならない（shall）。
8. When ユーザーがフローティングボタン（FAB）をタップしたとき、ホームナビゲーションは新規投稿モードのコンポーザー画面を表示しなければならない（shall）。

### Requirement 3: メモ削除
**Objective:** ユーザーとして、不要になったメモを取り除きたい。それによって、タイムラインを意図した内容に保てる。

#### Acceptance Criteria
1. When ユーザーがメモに対し削除操作（スワイプ等）を行ったとき、アプリは削除前に確認ダイアログを表示しなければならない（shall）。
2. If ユーザーが確認ダイアログで削除を取り消したとき、then アプリは対象メモを保持しなければならない（shall）。
3. When ユーザーが返信を持たない通常メモの削除を承認したとき、アプリは当該メモを物理削除し、その `likedBy` 関連を解除しなければならない（shall）。
4. When ユーザーがスレッドの親メモの削除を承認したとき、アプリは当該メモに紐づくすべての返信（自分・ペルソナを問わず）を連鎖削除（cascade）しなければならない（shall）。

### Requirement 4: ハッシュタグ抽出とタグ管理
**Objective:** ユーザーとして、本文に書いた `#タグ` で自動的に分類したい。それによって、後からテーマ別に振り返れる。

#### Acceptance Criteria
1. When メモが保存されるとき、アプリは本文を走査し、`#` の直後に続く文字・数字・アンダースコア（Unicode 文字クラス `\p{L}\p{N}_`）の連なりをタグとして抽出しなければならない（shall）。
2. The タグ抽出は日本語を含む非ラテン文字のタグ（例：`#メモ`）を抽出できなければならない（shall）。
3. When 抽出したタグ名と同名の `Tag` が既に存在するとき、アプリは既存の `Tag` を再利用しなければならない（shall）。
4. If 抽出したタグ名と同名の `Tag` が存在しないとき、then アプリは新しい `Tag` を作成しなければならない（shall）。
5. The アプリは抽出・解決した各 `Tag` を当該メモの `tags` に紐づけなければならない（shall）。
6. The `Tag` の `name` は先頭の `#` を含まず、同名の重複を持ってはならない（shall not）。

### Requirement 5: 検索とタグ絞り込み
**Objective:** ユーザーとして、本文やタグでメモを探したい。それによって、目的の記録に素早くたどり着ける。

#### Acceptance Criteria
1. When ユーザーが検索画面で文字列を入力したとき、検索機能は `body` に当該文字列を部分一致（contains）で含むメモを抽出しなければならない（shall）。
2. The 検索画面は登録済みタグの一覧を表示しなければならない（shall）。
3. When ユーザーがタグ一覧からタグを選択したとき、検索機能は当該タグに紐づくメモのみに結果を絞り込まなければならない（shall）。
4. The 検索結果は `createdAt` の降順（逆時系列）で並べなければならない（shall）。

### Requirement 6: スレッド（返信と親子表示）
**Objective:** ユーザーとして、あるメモに返信を連ねて文脈を追いたい。それによって、一連の流れをまとまりとして読める。

#### Acceptance Criteria
1. The スレッド詳細画面は親メモを最上部に表示しなければならない（shall）。
2. The スレッド詳細画面は親メモへの返信群を `createdAt` の昇順で表示しなければならない（shall）。
3. Where 返信がペルソナによるもの（`author != nil`）の場合、スレッド詳細画面はペルソナ名とアバター（`accentColor` で識別）を添えて自分の返信と区別しなければならない（shall）。
4. When ユーザーがスレッド詳細画面で「返信」を実行したとき、ホームナビゲーションは親メモIDを渡した返信モードのコンポーザー画面を表示しなければならない（shall）。

### Requirement 7: 画像添付
**Objective:** ユーザーとして、メモに画像を1枚添えたい。それによって、視覚的な記録を残せる。

#### Acceptance Criteria
1. When ユーザーがコンポーザーで `PhotosPicker` から画像を選択したとき、コンポーザーは選択画像を `Data` に変換し、保存するメモの `imageData` に格納しなければならない（shall）。
2. The コンポーザーは1件のメモにつき1枚までの画像添付に制限しなければならない（shall）。
3. Where 画像が添付されている場合、コンポーザーは投稿前にプレビューと添付削除ボタンを表示しなければならない（shall）。
4. The アプリは画像データを `@Attribute(.externalStorage)` 相当の外部保存により永続化しなければならない（shall）。

### Requirement 8: AIペルソナ
**Objective:** ユーザーとして、性格の異なる複数の AI キャラクターから反応を受けたい。それによって、一人のメモにも SNS のような手触りが生まれる。

#### Acceptance Criteria
1. When アプリが初回起動されたとき、アプリは性格（`personality`）の異なる複数のプリセットペルソナをシードしなければならない（shall）。
2. The 各ペルソナは表示名（`name`）・性格指示文（`personality`）・識別用の表示色（`accentColor`）を保持しなければならない（shall）。
3. The アプリはペルソナを AI キャラクターであると判別できる形で提示し、実在の他者を装ってはならない（shall not）。
4. When ペルソナが反応を生成するとき、アプリは当該ペルソナの `personality` を `instructions` とする独立したセッションを用いなければならない（shall）。

### Requirement 9: AIいいね・AIリプライ（構造化出力・非同期生成）
**Objective:** ユーザーとして、投稿した後にペルソナからのいいねや返信が届いてほしい。それによって、書きっぱなしにならず反応のループが生まれる。

#### Acceptance Criteria
1. When 新規メモの保存後に AI リアクション生成が開始されたとき、ペルソナエンジンは「いいねするか」「返信するか」「返信本文」を含む構造化された反応（`@Generable`）を生成しなければならない（shall）。
2. If 反応の「いいねする」が真のとき、then ペルソナエンジンは当該ペルソナを対象メモの `likedBy` に追加しなければならない（shall）。
3. If 反応の「返信する」が真のとき、then ペルソナエンジンは `author` を当該ペルソナ・`parent` を対象メモとする新しい返信メモを作成しスレッドへ挿入しなければならない（shall）。
4. The ペルソナエンジンは AI リアクションをバックグラウンドで生成し、投稿操作中に UI をブロックしてはならない（shall not）。
5. The ペルソナエンジンは複数ペルソナの反応を遅延を挟んで時間差で反映しなければならない（shall）。
6. While 複数の反応を生成している間、ペルソナエンジンはオンデバイスモデルへのセッションを逐次または低い並行度で実行しなければならない（shall）。

### Requirement 10: 関連メモの提示（RAG）
**Objective:** ユーザーとして、新しい投稿に関連する過去メモの気づきを得たい。それによって、自分の記録同士のつながりに気づける。

#### Acceptance Criteria
1. When 新規メモが投稿されたとき、RAG 機能は過去メモから関連するメモを検索しなければならない（shall）。
2. The RAG 機能が提示する「気づき」は、実際に存在する過去メモに基づいて（grounded）生成されなければならない（shall）。
3. Where Spotlight 検索ツールが利用可能な端末（iOS 27+）の場合、RAG 機能は各メモを Core Spotlight にインデックスし、Spotlight 経由で関連メモを取得しなければならない（shall）。
4. Where Spotlight 検索ツールが利用できない場合、RAG 機能はタグ重複スコアまたは本文の埋め込み類似度による代替経路で関連メモを抽出しなければならない（shall）。
5. When 関連メモが見つかったとき、スレッド詳細画面はそれをペルソナの「気づき」として提示し、参照元メモへ遷移できるようにしなければならない（shall）。

### Requirement 11: アクティビティ画面
**Objective:** ユーザーとして、誰がいつどの投稿に反応したかをまとめて見たい。それによって、届いた反応を見落とさず受け取れる。

#### Acceptance Criteria
1. Where AI 機能が有効な場合、アプリはアクティビティタブを表示しなければならない（shall）。
2. The アクティビティ画面はペルソナのリアクション（いいね・リプライ・気づき）を新しい順に一覧表示しなければならない（shall）。
3. When ユーザーがアクティビティの行をタップしたとき、アプリは該当メモまたはスレッドへ遷移しなければならない（shall）。

### Requirement 12: AI可用性に応じた縮退動作
**Objective:** ユーザーとして、AI 非対応の端末でも問題なくメモアプリを使いたい。それによって、端末を問わずコア体験が成立する。

#### Acceptance Criteria
1. When アプリが起動したとき、アプリは `SystemLanguageModel`（オンデバイスAI）の可用性を確認しなければならない（shall）。
2. If オンデバイス AI が利用不可のとき、then アプリはアクティビティタブと AI リアクション機能を非表示にしなければならない（shall）。
3. While AI が利用不可の状態でも、アプリはメモ投稿・タイムライン・削除・タグ・検索・スレッド・画像添付のコア機能を提供しなければならない（shall）。

### Requirement 13: ローカル完結とプライバシー
**Objective:** ユーザーとして、自分のメモが端末外に出ないことを保証してほしい。それによって、安心して個人的な内容を書ける。

#### Acceptance Criteria
1. The アプリはすべてのメモ・ペルソナ・タグ・画像データを端末のサンドボックス内に永続化しなければならない（shall）。
2. The アプリはネットワーク接続なしに、AI を含むすべての機能を動作させなければならない（shall）。
3. The アプリは AI 推論を端末内で完結させ、ユーザーデータを端末外へ送信してはならない（shall not）。
4. The アプリは外部サーバーやクラウドとの明示的な同期を行ってはならない（shall not）。

### Requirement 14: パフォーマンス
**Objective:** ユーザーとして、メモが大量になっても快適に使いたい。それによって、長期利用しても体験が劣化しない。

#### Acceptance Criteria
1. While メモが数千件規模に達している間でも、タイムラインは滑らかなスクロールを維持しなければならない（shall）。
2. The タイムラインおよび検索のソート・フィルタはデータストア側で実行しなければならない（shall）。
3. The AI 生成処理は UI スレッドをブロックしてはならない（shall not）。

### Requirement 15: プロダクト方針（あくまで X 風・他者要素を持たない）
**Objective:** ユーザーとして、Tegaru が「X のクローン」ではなく自分一人のためのメモ帳であってほしい。それによって、SNS の煩わしさなく X 風の手触りだけを得られる。

#### Acceptance Criteria
1. The Tegaru は X のロゴ・ブランド要素・固有の配色や UI レイアウトを模倣・流用してはならない（shall not）。
2. The Tegaru はフォロー・リツイート・外部通知・他ユーザーとの交流など、実在の他者が関与する SNS 機能を提供してはならない（shall not）。
3. The Tegaru が再現する「X 風」は、短文投稿・逆時系列の振り返り・反応が返るループという入力・閲覧・反応の流儀に限定されなければならない（shall）。
4. The Tegaru におけるすべての反応は端末内の AI ペルソナによって生成され、アプリ内で完結しなければならない（shall）。
5. While 反応を提示する間でも、Tegaru はそれが AI ペルソナによるものであることを明示し、実在の他者からの反応と誤認させてはならない（shall not）。

### Requirement 16: メモ編集
**Objective:** ユーザーとして、投稿済みのメモを後から直したい。それによって、削除・再投稿の手間なく内容を整えられる。

#### Acceptance Criteria
1. When ユーザーが既存メモの編集を開始したとき、コンポーザーは当該メモの現在の本文と添付画像を初期値として読み込んだ編集モードで開かなければならない（shall）。
2. When ユーザーが編集モードで保存を実行したとき、アプリは当該メモの `body` を更新し、画像が差し替え・削除された場合は `imageData` も更新しなければならない（shall）。
3. When 編集が保存されたとき、アプリは当該メモの `updatedAt` を現在時刻に設定しなければならない（shall）。
4. The アプリはメモ編集時に `createdAt` を変更してはならない（shall not）。
5. When 編集が保存されたとき、アプリは更新後の本文からタグを再抽出し、当該メモの `tags` を新しい抽出結果へ更新しなければならない（不要になったタグ関連は解除する）（shall）。
6. Where メモが一度以上編集されている（`updatedAt` が存在し `createdAt` より後の）場合、UI は当該メモに「編集済み」であることを示す表示を行わなければならない（shall）。
7. When ペルソナが既に反応したメモが編集されたとき、アプリは AI リアクションを再生成してはならず、既存の `likedBy` および返信を維持しなければならない（shall not / shall）。
